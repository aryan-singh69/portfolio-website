Movie Recommendation System

Content-based recommendation engine using TF-IDF vectorization and cosine similarity on movie metadata — generating personalized suggestions from the TMDB 5000 Movies dataset.

Overview

Problem

Users browsing large movie catalogs struggle to discover relevant titles. Simple genre filters miss nuanced content relationships — movies can share similar themes, directors, or cast members without overlapping genres. Manual browsing is time-consuming and misses non-obvious connections.

Dataset

TMDB 5000 Movies dataset: ~4800 movies after cleaning, with metadata including title, overview (plot summary), genres, keywords, top cast (up to 3), and director. Two CSV files (movies + credits) merged on title for a unified feature set.

Approach

Content-based filtering: combined movie metadata into a single text representation ("tags"), vectorized using TF-IDF (max 5000 features), and computed pairwise cosine similarity across all movies. Top-N most similar titles returned as recommendations.

System Architecture

The system is a batch-computed similarity engine. All heavy computation (vectorization, similarity matrix) happens once during build time. At query time, the system performs a simple lookup — no real-time ML inference required.

Feature Engineering

Genres, keywords, top 3 cast members, and director extracted from JSON-like metadata columns. Multi-word names collapsed (e.g., "Sam Worthington" → "samworthington") to prevent false token matches during TF-IDF. All features concatenated into a single "tags" string per movie.

Similarity Engine

TF-IDF vectorizer converts tags into 5000-dimensional sparse vectors. Cosine similarity computed between all pairs, producing a 4800×4800 similarity matrix (~23M pairwise scores). Matrix precomputed and stored — queries are index lookups, not recomputations.

Ranking & Output

For a given movie, its row in the similarity matrix is sorted in descending order. Top 5 entries (excluding itself) returned as recommendations. Results are deterministic — same input always produces same output since the matrix is static.

System Pipeline

1

Data Ingestion & Merging

Loaded tmdb_5000_movies.csv and tmdb_5000_credits.csv. Merged on movie title. Dropped rows with missing overview or genres. Final dataset: ~4800 movies with complete metadata.

2

Feature Extraction

Parsed JSON-encoded columns (genres, keywords, cast, crew) using ast.literal_eval. Extracted genre names, keyword terms, top 3 cast members, and director name. Each field cleaned: lowercased, spaces removed from multi-word names.

3

Tag Construction

Concatenated overview, genres, keywords, cast, and director into a single "tags" column per movie. Applied stemming (PorterStemmer) to reduce words to root forms ("loving" → "love", "action" → "action") for better matching.

4

Vectorization

Applied sklearn.feature_extraction.text.CountVectorizer (or TF-IDF) with max 5000 features and English stop words removed. Each movie represented as a sparse vector in 5000-dimensional term space.

5

Similarity Matrix Computation

Computed cosine similarity between all movie vectors using sklearn.metrics.pairwise.cosine_similarity. Result: 4800×4800 matrix stored in memory (~180MB). Build time: ~3–5 seconds on standard hardware.

6

Recommendation Query

User provides a movie title → system finds its index → retrieves its similarity row → sorts descending → returns top 5 titles with similarity scores. Query time: <50ms (pure index lookup + sort).

Qualitative evaluation: Tested manually on ~30 well-known movies. Franchise films (Batman, Avengers, Lord of the Rings) consistently return sequels and related entries. Genre-aligned recommendations are accurate for action, comedy, and drama. Weaker for niche or poorly-tagged movies with sparse metadata.

Evaluation note: Content-based systems without user interaction data cannot be evaluated with standard IR metrics (precision@K, NDCG) in a meaningful way. The recommendations were validated manually by checking thematic and cast/director coherence. No A/B testing was performed.

Impact

NLP Pipeline Practice

End-to-end text processing: messy JSON metadata → feature engineering → stemming → vectorization → similarity computation. A practical exercise in converting unstructured metadata into a usable vector space for information retrieval.

Recommendation Fundamentals

Demonstrates the core mechanics of content-based filtering — the same foundational approach used (alongside collaborative and hybrid methods) in production recommendation systems, scaled to much larger catalogs.

Extensible Foundation

The pipeline can be extended with collaborative filtering (user-item interaction matrix), embedding-based similarity (sentence transformers instead of TF-IDF), or hybrid approaches combining content and behavioral signals.

Challenges & Learnings

Metadata Sparsity

~15% of movies had incomplete metadata (missing keywords, empty cast lists, or generic overviews). These produced weak similarity signals. Dropping them entirely would reduce catalog size; keeping them adds noise. Chose to keep with minimum-field requirements.

Name Collision

Without collapsing multi-word names ("Sam Worthington" → "samworthington"), TF-IDF treated "Sam" and "Worthington" as separate tokens — creating false matches with other actors named Sam. Simple preprocessing fix with significant impact on result quality.

Cold Start Problem

Movies not in the TMDB dataset cannot be recommended. The similarity matrix is precomputed and static. Adding new movies requires rebuilding the entire matrix — no incremental update path in the current design.

Popularity Bias

Popular movies have richer metadata (longer overviews, more keywords, well-known cast), which produces stronger TF-IDF signals. Niche or indie films with minimal descriptions get weaker similarity scores regardless of actual content relevance.

Diversity Problem

Cosine similarity naturally favors highly similar items. Recommendations for franchise films return only sequels and spinoffs — no serendipitous discovery. Adding a diversity penalty (e.g., MMR — Maximal Marginal Relevance) was considered but not implemented.

Stemming Trade-offs

PorterStemmer sometimes over-stems: "universe" and "university" both stem to "univers", creating false matches. Tested without stemming — results were slightly worse for general queries but better for proper nouns. Kept stemming as default with awareness of edge cases.

Known Limitations

No User Personalization

Content-based only — no user interaction history, ratings, or watch data. Everyone searching the same movie gets identical results. True personalization requires collaborative filtering, which needs user-item interaction data not available here.

Static Catalog

The similarity matrix is computed once. No online learning, no incremental updates. If new movies are added to the dataset, the entire pipeline must be re-run — vectorization, matrix computation, and storage.

No Quality Signal

The system recommends similar movies, not good movies. A poorly-rated sequel still ranks high because it shares cast, director, and keywords with the original. No rating or review sentiment is factored into similarity.

Memory Footprint

The full 4800×4800 similarity matrix occupies ~180MB in memory. For production-scale catalogs (100K+ titles), this approach doesn't scale. Approximate nearest neighbor (ANN) methods (FAISS, Annoy) would be needed instead.
