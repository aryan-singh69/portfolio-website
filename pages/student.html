Student Result Prediction System

End-to-end machine learning pipeline for predicting student exam scores based on demographic and academic features — trained with 7 regression models and deployed via Flask.

Overview

Problem

Educational institutions want to identify students at risk of underperformance before final exams. Early prediction based on available data — demographics, preparation status, and prior scores — can enable timely interventions like tutoring or counseling.

Dataset

"Students Performance in Exams" dataset (~1000 records) with 8 features: gender, race/ethnicity, parental education level, lunch type (standard/reduced), test preparation course completion, and scores in math, reading, and writing.

Approach

Treated as a regression problem: predict math score (0–100) from remaining features. Trained 7 models (Linear Regression, Ridge, Lasso, Random Forest, Gradient Boosting, SVR, KNN), compared on R² and MAE, deployed the best via Flask.

System Architecture

The project is structured as a modular ML pipeline with separate components for data ingestion, transformation, model training, and prediction — each independently testable and reusable. The architecture follows a standard sklearn pipeline pattern.

Data Ingestion Module

Handles CSV loading, validation, and train/test splitting (80/20). Checks for missing values and expected column types. Outputs raw DataFrames for downstream processing. Designed to be swappable if data source changes (e.g., database instead of CSV).

Transformation Pipeline

Sklearn ColumnTransformer combining: StandardScaler for numerical features (reading_score, writing_score) and OneHotEncoder for categorical features (gender, ethnicity, parental_education, lunch, test_prep). Single pipeline object handles all preprocessing.

Model + Serving Layer

Best model and preprocessor serialized together as a pickle file. Flask app loads the pipeline on startup. Prediction requires only raw feature input — the pipeline handles transformation internally. No separate preprocessing step at inference time.

ML Pipeline

1

Data Ingestion

Raw CSV loaded with Pandas. Validated column names and types. Split into train/test sets (80/20) using train_test_split with random_state=42 for reproducibility. No stratification — continuous target variable.

2

Exploratory Analysis

Correlation analysis: reading_score and writing_score are strongly correlated with math_score (r ≈ 0.82, 0.80). Distribution of math scores is roughly normal (mean ~66, std ~15). Lunch type and test prep show measurable effect on scores.

3

Feature Transformation

Numerical features scaled to zero-mean, unit-variance via StandardScaler. Categorical features one-hot encoded (5 categorical columns → ~15 binary features after encoding). Combined using ColumnTransformer for a clean preprocessing pipeline.

4

Multi-Model Training

Trained 7 regression models on the same preprocessed data: Linear Regression, Ridge (α=1.0), Lasso (α=1.0), Random Forest (100 trees), Gradient Boosting (100 estimators), SVR (RBF kernel), and KNN (k=5). All trained with default or lightly-tuned hyperparameters.

5

Evaluation & Selection

All models evaluated on held-out test set using R² score and MAE. Ridge Regression selected as best model (~87% R², MAE ~5.2). Cross-validated with 5-fold CV to confirm generalization. Model checkpoint saved at best performance.

6

Flask Deployment

Web interface built with Flask + Jinja2 templates. User fills a form with student details → POST request sends data to prediction endpoint → pipeline preprocesses input and returns predicted math score → result displayed on response page.

Impact

Pipeline Engineering

The main learning is building a modular ML pipeline with separate, reusable components for ingestion, transformation, training, and prediction. Each component can be tested and modified independently — a pattern used in production ML systems.

Model Deployment Practice

Demonstrates the full path from training notebook to a working web application: sklearn pipeline serialization, Flask endpoint creation, HTML form design, and end-to-end request-response flow. The deployment is functional, not just theoretical.

Systematic Evaluation

Comparing 7 models with consistent metrics (R², MAE, cross-validation) and selecting based on evidence rather than intuition. Documents the reasoning behind model selection — a fundamental skill in applied ML work.

Challenges & Learnings

Feature Leakage Risk

Reading and writing scores are highly correlated with math score (r ≈ 0.8). In a real early-warning system, these might not be available at prediction time if all exams happen simultaneously. The model works more like a score correlator than a true predictor of at-risk students.

Small Dataset Effects

~1000 records is borderline for training complex models. Random Forest and Gradient Boosting showed overfitting symptoms (train R² ~0.97 vs test R² ~0.85). Linear models generalized better on this data size, which is expected.

Categorical Encoding Explosion

OneHotEncoding 5 categorical features with multiple levels produced ~15 binary columns from ~5 original features. With only 1000 rows, this increases the feature-to-sample ratio. Tested target encoding as an alternative but saw minimal difference.

Hyperparameter Tuning

Used mostly default hyperparameters. Attempted GridSearchCV for Random Forest (n_estimators, max_depth) but gains were marginal (<0.5% R²) given the small dataset. Concluded that data size, not model complexity, was the bottleneck.

Serialization Issues

Initial attempt to pickle the ColumnTransformer failed with custom preprocessing steps. Switched to dill for serialization, which handles lambda functions and nested objects that standard pickle cannot.

Deployment Validation

Predictions from the Flask app initially differed from notebook results due to inconsistent feature ordering in the form data. Fixed by explicitly mapping form fields to the expected column order before feeding to the pipeline.

Known Limitations

Feature Leakage

Reading and writing scores dominate the prediction. Without them, R² drops to ~0.25 — meaning the demographic features alone are weak predictors. This limits real-world applicability for early warning scenarios.

No Fairness Analysis

The model uses gender and ethnicity as input features, which could introduce or reinforce bias. No disparate impact testing, calibration analysis, or fairness audit was performed. This would be critical for any real educational deployment.

Static Model

Deployed model is a saved pickle — no retraining pipeline, no monitoring, no drift detection. Student data distributions change year-over-year (different demographics, curriculum changes), and the model would need periodic updates.

Synthetic-Feeling Dataset

The "Students Performance" dataset is commonly used in tutorials and may not reflect real institutional data complexity. Real student data would have many more features, more noise, and stricter privacy requirements (FERPA, GDPR).
