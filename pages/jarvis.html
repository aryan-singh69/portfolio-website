Jarvis — AI Assistant System

Modular local AI assistant for real-time task execution and system automation — built with a plugin-based architecture for extensible command processing via text and voice inputs.

Overview

Problem

Most AI assistants are cloud-dependent and closed-source, offering no customization or local control. For developers who want to automate repetitive desktop tasks — opening apps, running searches, fetching system info — there's no lightweight, extensible solution that works entirely offline.

Input

User commands via two channels: text input (CLI) and voice input (microphone, processed through Google Web Speech API). Commands are natural-language-like but follow a keyword-first pattern (e.g., "open chrome", "search python tutorials", "weather Delhi").

Approach

Plugin-based Python application with keyword routing. Each task type (app control, web search, media playback, system info) is handled by an independent module. No ML models — intentionally deterministic. Focus is on architecture, API integration, and extensibility.

System Architecture

The system follows a modular pipeline: input is captured, parsed against a routing table, dispatched to the appropriate execution module, and the result is returned as text or speech. Each module is independent — adding a new capability means adding a new file.

Input Layer

Dual-mode input handler: CLI text input (immediate) or microphone capture via speech_recognition library. Voice is converted to text string using Google Web Speech API before entering the same processing pipeline as text input.

Command Router

Keyword-based routing table mapping trigger words to execution modules. First-match strategy: input string is scanned for registered keywords in priority order. No NLU, no intent classification — deterministic and predictable.

Execution Modules

Each module handles one task category: app_launcher (subprocess calls), web_search (browser automation), media_player (YouTube/local), system_info (datetime, battery, IP), knowledge (Wikipedia API).

System Pipeline

1

Input Capture

User input received via text (CLI prompt) or voice (microphone → speech_recognition → Google Web Speech API). Voice recognition returns a text transcript with ~1–2s latency depending on network. Fallback to text input on recognition failure.

2

Command Parsing & Intent Mapping

Input string converted to lowercase and scanned against ~20 registered keyword patterns. Each pattern maps to a specific module and action. Example: "open" → app_launcher.launch(), "search" → web_search.query(), "play" → media_player.play(). Unmatched inputs trigger a default "I don't understand" response.

3

Module Execution

Matched module executes the task: opening applications via subprocess.Popen(), running web searches via webbrowser.open(), fetching data from APIs (OpenWeatherMap, Wikipedia), or performing OS-level operations (shutdown, restart, screenshot).

4

Response Generation

Module returns a result string. For informational queries (weather, time, Wikipedia), the result contains extracted data. For action queries (open app, play music), the result is a confirmation message.

5

Output Delivery

Result displayed as text in terminal and optionally spoken aloud using pyttsx3 (offline text-to-speech engine, no network required). The main loop continues until an exit command ("quit", "exit", "bye") is received.

Impact

Architecture Practice

The primary value is in the plugin-based design pattern. Each capability is a standalone module with a consistent interface — demonstrating how to structure extensible Python applications where features can be added independently without modifying core logic.

API Integration

Practical experience integrating 5+ third-party APIs (speech recognition, text-to-speech, weather, Wikipedia, YouTube) into a single cohesive application with error handling, timeout management, and fallback behavior.

Automation Workflow

Demonstrated system-level automation on Windows: launching applications, controlling media playback, capturing screenshots, and executing OS commands — all triggered from a unified command interface.

Challenges & Learnings

Ambiguity Handling

Keyword routing cannot resolve ambiguous commands. "Play something" doesn't specify what to play. Addressed by requiring specific arguments after the keyword, but this makes the interface less natural than conversational assistants.

Voice Recognition Noise

Google Web Speech API accuracy degrades in noisy environments and with non-native accents. Misrecognized words cascade into wrong module routing. Added a confirmation step for destructive commands (shutdown, restart) to prevent accidental execution.

API Rate Limits

OpenWeatherMap free tier allows 60 calls/minute. Wikipedia API has soft rate limits. Implemented basic caching (in-memory dict) for repeated queries within the same session to reduce redundant API calls.

Cross-Platform Issues

Subprocess calls for app launching use Windows-specific paths and commands (os.startfile, taskkill). System is not portable without writing OS-detection logic and platform-specific module variants.

Stateless Design

Each command is independent — no session memory, no conversation context. Follow-up queries ("do that again", "what about Mumbai?") are not supported. Adding a context stack was considered but out of scope for this version.

Error Recovery

Network failures during API calls initially crashed the main loop. Added try/except wrappers around all API calls with user-facing error messages ("Couldn't fetch weather — check your internet") instead of silent failures.

Known Limitations

No Natural Language Understanding

The system uses keyword matching, not intent recognition. Conversational or indirect commands fail. "Can you maybe open Chrome for me?" would not reliably trigger the app launcher.

No Context / Memory

Each command is stateless. The system cannot handle multi-turn interactions, follow-up questions, or references to previous queries. Every input is processed independently.

Windows Only

Built and tested exclusively on Windows. App launching, file system paths, and system control commands are OS-specific. Would require rewriting execution modules for macOS/Linux.

Not a Production Assistant

This is a learning project demonstrating modular design and API integration — not a replacement for production voice assistants. No wake word, no continuous listening, no sophisticated NLU.
